# Bugs & Issues Log

Tracks bugs found in the legacy `100UP_suite_V46.html` calculator suite and
the CRM app, whether they've been carried into the new build, and their
resolution status.

| # | Tool / Screen | Issue | Resolution | Status |
|---|---|---|---|---|
| 1 | Simulation Trace (V46) | Typing `0` into Daily load, Number of panels, or Battery kWh silently reverts to the hardcoded default (20 / 30 / 27) instead of using 0. Caused by `Number(value) \|\| default` — `0` is falsy in JS so the fallback always wins. Symptom: table stays at "Load (kWh) 20" even after zeroing the field, and results reflect the default scenario, not zero. Standby W is unaffected (its fallback is also 0). | Patched in `100UP_suite_V46.html` (`runSimTrace()`) to use a proper finite-number check instead of `\|\|`. Will be built correctly from scratch in the CRM port (no `\|\|`-on-numeric-input pattern). | Fixed (V46) / To-build-correctly (CRM port) |
| 2 | Simulation Trace → "Quick-fill from assumptions" (V46) | The button labels/kWh values are a hybrid: battery kWh figures are pulled live from `A.sigBatteryKwh` / `A.deyeBatteryKwh`, but the panel counts (30/36) and which unit-count combos are offered (Sig 3/4, Deye 5/6) are hardcoded literals in the JS — not derived from assumptions at all. Editing assumptions only changes the kWh numbers shown, not which configs exist. | Not fixed — documented behaviour. To be redesigned properly when Simulation is ported (quick-fill configs should probably become data-driven, e.g. stored alongside assumptions or computed from available battery/inverter combos). | Open — design decision needed at port time |
| 3 | Assumptions table → "Stock avail." column (V46) | The Stock avail. column looks up on-hand stock **quantity** for a given assumption row (matched by stock name, or by panel wattage for `panelCost`), but the assumption's **cost** value (e.g. `panelCost`, `sigInverterCost`, `deyeBatteryCost`) is a separately hand-maintained number with no link back to the actual purchase/supply cost recorded against that stock item (via receipts). Qty and cost can silently drift out of sync — you see live stock availability next to a cost figure that isn't tied to it. | Partially fixed: `stocks` now carries `category`/`manufacturer`/`model`/`watts`/`kw`/`kwh`/`kva`/`last_cost` directly (merged with the old `stock_ces_specs` table, migration `20260719020001_merge_stock_specs.sql`), so a real cost field exists on the stock record. Assumptions still isn't wired to it — that's the next step, deferred until Assumptions gets a `stock_id` FK per cost field (see #4). `last_cost` is also still hand-edited on the Stock page; Receive Stock doesn't capture per-line cost yet (see #4). | Open — schema half-done, linkage deferred |
| 4 | Stock / Receive Stock / Assumptions — no Purchase Order flow | `receive_stock` RPC has no unit-cost input, so "receiving stock" never records what you actually paid. There's also no concept of an open order (sent → partially received → closed) — `OrderList` only tells you what's short, it doesn't produce a trackable PO. | Partially fixed: `receipts` renamed to `purchase_orders` with `po_ref`/`po_amount`/`po_status` (sent/partially_received/closed) + new `purchase_order_items` (qty_ordered, cost, qty_received) — migration `20260719030001_purchase_orders.sql`. `create_purchase_order` RPC + `OrderList` "💾 Save PO" button let you save a short-item group as a real, trackable PO. New `Purchase Orders` page (list + detail, read-only) shows them. Still open: no "receive against an open PO" flow yet — `receive_stock` (ad-hoc paste-invoice, Stock page) still creates its own already-`closed` PO with no unit cost captured, unchanged from before. Receiving against a `sent`/`partially_received` PO (updating `qty_received` + `stocks.last_cost` per line, advancing `po_status`) and the Suppliers page PO-list rework are the next steps. | Open — schema + save/view done, receive-against-PO deferred |
| 5 | Assumptions ↔ Stock linkage — design decision (not yet built) | Traced the full calculator → CRM pipeline (`buildQuoteBom`/`partInverter`/`partBattery` in V46, `LinkQuoteModal`/`normalizePart` in the CRM). Confirmed: there is **no hidden linkage** anywhere — assumption cost fields (`sig_battery_kwh`, `deye_inverter_cost`, etc.) only drive sizing/pricing math; the result is turned into a **generated name** via hardcoded JS templates (`SigenStor EC ${kw}.0 ${SP\|TP}`, `Deye AI-W5.1-${kw}${P1\|P3}-AU-B`); the CRM then fuzzy-matches that name string against `stocks.name` via `normalizePart()` regex. Swapping a brand/model means changing the naming template *and* the regex *and* (for a genuinely new brand) adding new assumption columns — three places, not one. | Design direction agreed, not built: (a) **Panel** (`panel_cost`/`panel_w`/`panel_mfr`/`panel_model`) is the clean case — single item, single cost/spec — add a nullable link + a manual "refresh" action that pulls `last_cost`/`watts`/`manufacturer`/`model` from the linked `stocks` row into the assumption fields (assumption stays an editable input, not a live join — this is also the exact gap the calculator already flags via `mapNeeded: true` on the panel BOM line). (b) **Sigenergy/Deye kit families** are NOT single-item links — they're naming templates picked by a kW/kWh threshold. Proper fix there is replacing `normalizePart()`'s hardcoded regex with an editable `canonical_key → stock_id` alias table, so EOL/brand swaps become a data edit for existing tiers — though adding genuinely new tiers or a third brand family still needs a code + schema change regardless. No table schema chosen yet (leaning mapping-table style per bug's earlier note, still open). | Open — captured, no build yet |

## Notes
- V46 file: `100UP_suite_V46.html` (legacy standalone HTML calculator suite, `localStorage`-persisted assumptions under key `100upSolarCalc_v24`).
- CRM port: `app/src/pages/*` (new Supabase-backed app). Assumptions are being migrated to a Supabase table (see `supabase/migrations/`) so Simulation, Daily Load Profile, and other Quote Designer tools can read live data instead of `localStorage`.
