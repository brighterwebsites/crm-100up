## Current schema (for reference â€” see migrations for full detail)

Tables: `pipeline_steps`, `profiles`, `jobs`, `job_stock_items`, `stocks`,
`suppliers`, `receipts`, `stock_ces_specs`, `job_events`.

`jobs` columns today: `id, name, location, system_description, value, email,

date_booked, install_start, install_date, ces_submitted, ces_received,
rebate_submitted, rebate_received, fixes_needed, job_order (jsonb), version,
created_at, updated_at`.

Stage/step transitions and the pipeline-linked date columns are gated by a
`SECURITY DEFINER` RPC set (`advance_job_stage`, `move_job_back`,

(`private.guard_jobs_update()` in `supabase/migrations/20260718090003_functions_triggers.sql`)

outside those functions. This is the fix for a documented bug in the old app
(editing a date field directly bypassed stock consumption) â€” any change
touching these columns must preserve that guarantee.

## 1. Split `customers` out of `jobs` â€” AGREED

**Problem.** `jobs` embeds customer contact fields (`name`, `email`, `phone`,
`contact_method`) directly on every job row, with no relationship linking two
jobs to the same customer â€” not even by convention, even though `job_type =
'service'` already represents a returning customer's second job.

**Proposed schema.**

```sql
customers
  id             bigint generated by default as identity primary key
  name           text   not null
  address        text   not null default ''   -- optional mailing/billing address;
                                              -- leave empty if only install site is known
  phone          text   not null default ''
  email          text   not null default ''
  contact_method text   not null default 'Email'
  version        integer not null default 1
  created_at     timestamptz not null default now()
  updated_at     timestamptz not null default now()
```

`jobs` changes:
- **Remove:** `name`, `email`, `phone`, `contact_method` (moved to `customers`)
- **Add:** `customer_id bigint not null references customers(id) on delete restrict`
- **Keep unchanged:** `location` â€” this is the install site address, deliberately
  distinct from `customers.address` (a customer's billing/contact address can
  differ from where a system is installed, e.g. rental property or business
  premises). Do not collapse these into one field.

**`customers.address` note.** The legacy export contains only the job's
install-site `loc` field; there is no separate mailing address in the source
data. Import `customers.address` as empty string for all rows. The UI should
expose it as an optional field rather than treating it as always populated.

**`customer_id` index.** Add `create index jobs_customer_id_idx on public.jobs
(customer_id)` â€” required for the RLS join and FK cascade performance.

**`customer_id` nullability.** The column is defined `NOT NULL` at migration
time. Because import runs before adding the constraint (or as a single atomic
import), this is safe; no orphan jobs can exist after cutover.

**`ON DELETE RESTRICT`.** Prevents deleting a customer who still has jobs â€”
consistent with the existing `job_stock_items.stock_id ... on delete restrict`
pattern. Admin must delete all a customer's jobs first.

**Customer lifecycle.** Three paths need explicit design in the app:

1. **New install job:** admin creates a new customer inline with the job.
   `createJob()` must become (or call) an RPC or two-step transaction that
   inserts a `customers` row then a `jobs` row with the resulting `customer_id`.
   `jobs.customer_id NOT NULL` enforces this â€” no job can exist without a
   customer.

2. **Service/upgrade job for returning customer:** admin selects an existing
   customer by name/phone lookup rather than re-entering contact details. The
   "Link calculator quote" modal already uses a match-by-name approach and is a
   natural model for this. Add PARKED for the UI design until the create-job
   flow is settled.

3. **Shared-customer edits:** editing customer contact fields (name, phone,
   email, contact_method) via one job should update `customers` directly, not
   a per-job copy. The job detail form must save customer fields to `customers`
   and job fields to `jobs` separately. Only admin can edit customer fields
   (same rule as today for those columns).

**`contact_method` note.** Today `contact_method` has no edit field in the
React UI â€” it is set only via import and appears only in the backup export.
Moving it to `customers` does not add regression risk, but the JobDetail form
should gain a `contact_method` selector when customer editing is built, so
the field is not permanently inaccessible.

**Version + triggers.** `customers.version` requires a bump trigger mirroring
`private.bump_job_version()`. Add `private.bump_customer_version()` in the
same migration, triggered `BEFORE UPDATE ON public.customers`. Same pattern for
`installation_requests` (Â§2).

**Migration consideration.** Existing trial job rows have customer data
embedded per-row with no identity linking. At cutover the import runs against
a **fresh export from the old app** â€” the importer must produce a dry-run
customer-match report (see Â§Execution plan, step 5) for human review before
writing any rows. Do not auto-merge silently.

**RLS.** New policies on `customers`:

```sql
-- Enable and grant
alter table public.customers enable row level security;
grant select, insert, update, delete on public.customers to authenticated;
grant usage, select on sequence public.customers_id_seq to authenticated;

-- Installers can read a customer reachable through a job assigned to them
create policy "customers_select_admin_or_via_job" on public.customers
  for select to authenticated
  using (
    (select private.is_admin())
    or exists (
      select 1 from public.jobs j
      where j.customer_id = customers.id
        and j.assigned_installer_id = (select auth.uid())
    )
  );

-- Admin-only writes
create policy "customers_insert_admin" on public.customers
  for insert to authenticated
  with check ((select private.is_admin()));

create policy "customers_update_admin" on public.customers
  for update to authenticated
  using ((select private.is_admin()))
  with check ((select private.is_admin()));

create policy "customers_delete_admin" on public.customers
  for delete to authenticated
  using ((select private.is_admin()));
```

Installer access is intentional â€” they need contact details to reach the
customer on install day. No customer-facing login exists, so no customer-owned
policy is needed.

## 2. `jobs.job_order` (jsonb) â†’ real `installation_requests` table â€” AGREED

**Problem.** `job_order` is an unstructured jsonb blob carried over from the
old app's shape: `{ref, issued, installer, vehicle, siteAccess,
specialInstructions, systemDesc, extraNotes, savedAt, customItems[]}`. No
cross-job query surface (e.g. "which vehicle was used most often").

**Proposed schema.** `job_id` is the primary key (not a surrogate identity)
to enforce 1:1 cardinality and mirror the `stock_ces_specs` extension-table
pattern â€” absence of a row is meaningful, presence is always unique.

```sql
installation_requests
  job_id                bigint primary key references jobs(id) on delete cascade
  job_order_ref         text   not null default ''
  issued_date           date
  vehicle               text   not null default ''   -- free text; see Â§5 (PARKED)
  site_access_notes     text   not null default ''
  special_instructions  text   not null default ''
  additional_notes      text   not null default ''
  custom_items          jsonb  not null default '[]'::jsonb
                                -- small ad hoc line-item list; not worth a child
                                -- table at this scale â€” revisit only if it needs
                                -- independent querying later
  version               integer not null default 1
  created_at            timestamptz not null default now()
  updated_at            timestamptz not null default now()
```

**`savedAt` â†’ `updated_at`.** The old blob's `savedAt` (epoch ms) served as
a last-saved timestamp. The `updated_at` column on the new table serves the
same purpose and is maintained by a trigger (see Version + triggers below).

Dropped from the old jsonb shape (superseded, not carried forward):
- `systemDesc` â€” duplicate of `jobs.system_description`
- `installer` (free-text name) â€” superseded by `jobs.assigned_installer_id`,
  a real FK to `profiles`. The `JobOrderModal` currently shows this field; it
  should be removed from the form and replaced with a display of the assigned
  installer's name from `profiles`.

**IMPORTANT â€” what does NOT move here.** `jobs.stage`, `jobs.step`,
`jobs.assigned_installer_id`, and all pipeline-gate dates (see Â§3) **stay on
`jobs`**, unchanged. Reasons:

- `assigned_installer_id` is what the installer-visibility RLS policy checks
  directly. Moving it to a child table turns every job-visibility check into
  a join.
- The guard-trigger machinery locks and updates dates on the same row it
  checks stage/step on, in one statement.
- `ces_submitted/received`, `rebate_submitted/received` are Stage 4
  compliance state, not installation logistics.

**Version + triggers.** Add `private.bump_installation_request_version()`
triggered `BEFORE UPDATE ON public.installation_requests`, mirroring the
`bump_job_version` pattern.

**RLS.** Same shape as `job_stock_items` today: visible to admin or to the
installer assigned to the parent job (via `jobs.assigned_installer_id`).
Direct writes admin-only. No RPC needed unless that changes.

```sql
alter table public.installation_requests enable row level security;
grant select, insert, update, delete on public.installation_requests to authenticated;

create policy "installation_requests_select_admin_or_assigned"
  on public.installation_requests
  for select to authenticated
  using (
    (select private.is_admin())
    or exists (
      select 1 from public.jobs j
      where j.id = installation_requests.job_id
        and j.assigned_installer_id = (select auth.uid())
    )
  );

create policy "installation_requests_insert_admin" on public.installation_requests
  for insert to authenticated
  with check ((select private.is_admin()));

create policy "installation_requests_update_admin" on public.installation_requests
  for update to authenticated
  using ((select private.is_admin()))
  with check ((select private.is_admin()));

create policy "installation_requests_delete_admin" on public.installation_requests
  for delete to authenticated
  using ((select private.is_admin()));
```

**Realtime.** Add `installation_requests` to `supabase_realtime` if the UI
should receive live updates when an admin saves a job order while an installer
has the job open:

```sql
alter publication supabase_realtime add table public.installation_requests;
```

## 3. Job-date renames for planned/actual clarity â€” AGREED

**Problem.** The three pipeline dates already form a planned â†’ actual-start â†’
actual-end sequence, but aren't named that way.

| Current name    | Set at step                    | Actually means            |
|-----------------|--------------------------------|---------------------------|
| `date_booked`   | (3,1) "Date booked"            | Planned install date       |
| `install_start` | (3,3) "Install in progress"    | Actual install start       |
| `install_date`  | (3,4) "Install complete"       | Actual install completion  |

A multi-day install is already representable â€” the job stays at "Install in
progress" until truly finished. The only real problem is that `install_date`
reads ambiguously. **Confirmed: no per-visit tracking is wanted** â€” a single
start/end span is sufficient.

**Proposed â€” pure rename, no new columns, columns remain PostgreSQL `date` type:**
- `date_booked`   â†’ `planned_install_date`
- `install_start` â†’ `install_start_date`
- `install_date`  â†’ `install_completion_date`

**Note on naming.** These are `date`-typed columns (not `timestamptz`).
Using a `_date` suffix rather than `_at` makes the type explicit at a glance;
`_at` conventionally implies `timestamptz` in Postgres conventions.

**Impact â€” every one of these needs updating for the rename:**

*Database (new migration):*
- `jobs` table column definitions
- `private.guard_jobs_update()` â€” protected-column list references these by
  name (currently lines 41â€“43 of `20260718090003_functions_triggers.sql`)
- `advance_job_stage()` â€” CASE expressions and date assignments
  (migration 003 lines 199â€“212)
- `move_job_back()` â€” date-clearing CASE expressions (migration 003 lines
  340â€“349)
- `reschedule_booking()` â€” direct `date_booked` write and event payload key
  (effective live body in **migration 004**, not 003)

*App (coordinated deployment):*
- `app/src/types/database.types.ts` â€” regenerate after migrating
- `app/src/lib/data.tsx` â€” `Job` type derives from generated types
- `app/src/lib/pipeline.ts` â€” `nextStepNeedsDate()` return union literals
  (`'date_booked'`, `'install_start'`, `'install_date'`) must change to new
  names; callers in `JobDetail.tsx` use these as labels
- `app/src/lib/stockCalc.ts` â€” `commitDate()` reads `j.date_booked` and
  `j.install_date`; `bookedInstalls()` reads `j.date_booked` and
  `j.install_date`
- `app/src/features/jobs/JobDetail.tsx` â€” date display labels and the
  `dateAsk.field` string comparison
- `app/src/features/jobs/JobsBoard.tsx` â€” `job.date_booked` and
  `job.install_date` card display
- `app/src/features/jobs/actions.ts` â€” `jobDetailsText()` and `buildCes()`
  date fields
- `app/src/features/stock/OrderList.tsx` â€” `job.date_booked || job.install_date`
  display
- `app/src/pages/Shell.tsx` â€” backup export maps `date_booked`,

## 7. `stock_ces_specs.manufacturer` â†’ real `manufacturers` table â€” PARKED

**Problem.** `stock_ces_specs.manufacturer` is free text, duplicated across
every stock item from the same maker. The one fact that actually matters here
â€” *"Deye and Jinko legal manufacturer names are verified; Sigenergy and Trina
are not"* (per `CLAUDE.md`) â€” is a fact about the **manufacturer's legal
name**, not about any individual stock item's spec row.

**Not in this round.** The `verified` column on `stock_ces_specs` is
per-spec-row (a Sigenergy 8 kWh and a Sigenergy 15 kVa inverter each carry
their own `verified = false` flag). Collapsing to a manufacturer-level
`cec_verified` flag would lose the ability to give per-model CEC warnings
in `buildCes()` without redesigning that function. The full manufacturer
normalization requires:

1. Deciding whether to keep per-spec `verified` alongside per-manufacturer
   `cec_verified`, or replace one with the other.
2. Answering what else, if anything, is stored per manufacturer (ABN, CEC
   listing URL, standards version).
3. Updating `buildCes()` in `app/src/features/jobs/actions.ts` to read the
   new shape.
4. Updating `scripts/import_from_export.py` to emit `manufacturers` rows and
   `stock_ces_specs.manufacturer_id`.

**Defer until all four questions are settled.** Do not include this in the
Phase 2 migration bundle.

## 8. Why two tables â€” `stocks` vs `stock_ces_specs` â€” explained, no change proposed

`stocks` is every inventory item, including things with no CES relevance at
all (mounting brackets, cable ties). `stock_ces_specs` is an optional 1:1
extension that only has a row for items that are actually CES-relevant
(inverters, batteries, panels) â€” the **absence** of a row is meaningful (it
drives the old app's "not in the CES catalogue" warning, preserved in
`app/src/features/jobs/actions.ts`'s `buildCes()`). Classic core-entity +
optional-specialized-extension pattern. No action here.

## Application contract â€” how the frontend must change

### Data types

`Job = Tables<'jobs'>` (currently from `app/src/lib/data.tsx`) will not contain
customer fields after the split. Introduce a composite type for any component
that needs customer data:

```typescript
export type Customer = Tables<'customers'>
export type InstallationRequest = Tables<'installation_requests'>

export interface JobWithRelations extends Job {
  customer: Customer
  installation_request?: InstallationRequest | null
}
```

The `DataContext` can either:
- Fetch `customers` in parallel with `jobs` and join client-side (simple,
  consistent with the current parallel-fetch pattern in `data.tsx`), or
- Use a Supabase embedded select: `supabase.from('jobs').select('*, customers(*)')`.

**Recommendation:** parallel fetch, same as today. The client-side join is
trivial at this data size and keeps the fetch pattern uniform.

### Create-job path

`createJob()` currently inserts `{ name }` into `jobs`. After the split:

1. Insert into `customers` â†’ get `customer_id`.
2. Insert into `jobs` with `customer_id`.

These must be atomic. Options:
- Admin-callable RPC `create_job_with_customer(name, phone, email, ...)`.
- Two-step with error handling that deletes the orphaned customer if the job
  insert fails.

The RPC approach is cleaner and consistent with the existing pattern.

### Save-details path

`updateJob()` currently patches `jobs` only. After the split, the admin save
must:
- Write customer fields (`name`, `phone`, `email`, `contact_method`,
  `address`) to `customers`.
- Write job fields (`location`, `system_description`, `value`, `job_type`,
  `assigned_installer_id`, `notes`, `fixes_needed`) to `jobs`.

Installer save path (notes + fixes only) is unaffected â€” neither field moves.

### Job order save path

`JobOrderModal` saves to `jobs.job_order`. After the split, it saves to
`installation_requests`. The save function must:
- Check if a row exists (via primary key `job_id`) and use `upsert`.
- Remove the `installer` and `systemDesc` fields from the form (they are
  superseded or duplicated elsewhere).

### Backup export (`Shell.tsx`)

The `exportJson()` function in `app/src/pages/Shell.tsx` produces a
legacy-compatible JSON file Fred uses as a hands-on backup. This must continue
to work and produce the same old-app-readable shape. After the split, the
export must denormalize:

```typescript
// current:        j.name, j.email, j.phone, j.contact, j.job_order
// after split:    customer.name, customer.email, customer.phone,
//                 customer.contact_method, installationRequest
```

Map the renamed date columns back to their legacy camelCase keys:
- `planned_install_date` â†’ `dateBooked`
- `install_start_date`   â†’ `installStart`
- `install_completion_date` â†’ `installDate`

Do not change the export format version; the old app must remain able to
import the file.

### Realtime

The `DataContext` subscribe covers all `public.*` changes. New tables
(`customers`, `installation_requests`) only fire the refresh if they are added
to the `supabase_realtime` publication. Add them if live updates are expected
(see Â§2 for the `installation_requests` publication statement; add `customers`
too if editing a shared customer from one browser should update another).

## Execution plan, once everything above is settled

This is a restructuring on the same scale as the original schema/RLS build.
Same checkpoint discipline applies.

**Scope of this round (Phase 2):** Â§1â€“Â§4 only.
