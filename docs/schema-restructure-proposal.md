# Schema Restructure Proposal — Customers, Installation Requests, Date Clarity, Notifications

**Status: proposed, not implemented.** Nothing in this document has been applied to
the database or reflected in the deployed app. This captures the outcome of a
design-review conversation so work can continue in any tool or session without
re-deriving the reasoning from chat history.

## How to use this document

- This describes changes to the schema **currently live** at Supabase project
  `nmyczgnvjhwhgpgvwfdx`. Ground truth for "what exists today" is
  `supabase/migrations/*.sql`, not this doc — cross-check before implementing
  anything below, in case migrations have moved on since this was written.
- The project is **pre-cutover**: the old `100UP_suite_V46.html` localStorage app
  remains the authoritative source of truth during the trial (see
  `/root/.claude/plans/superbase-is-going-to-woolly-book.md`), and the only
  consumer of this Postgres schema is the React app in `app/`. That means direct,
  in-place renames are safe and preferred over parallel-table migrations — there
  are no external consumers to coordinate with, and a rename can land as one
  migration + one frontend update, deployed together.
- Each item below is tagged **AGREED** (settled, ready to design/implement in
  detail), **PARKED** (discussed, deliberately deferred, do not implement), or
  **UNDECIDED** (open question, needs an answer before implementing).

## Current schema (for reference — see migrations for full detail)

Tables: `pipeline_steps`, `profiles`, `jobs`, `job_stock_items`, `stocks`,
`suppliers`, `receipts`, `stock_ces_specs`, `job_events`.

`jobs` columns today: `id, name, location, system_description, value, email,
phone, contact_method, job_type, stage, step, notes, assigned_installer_id,
date_booked, install_start, install_date, ces_submitted, ces_received,
rebate_submitted, rebate_received, fixes_needed, job_order (jsonb), version,
created_at, updated_at`.

Stage/step transitions and the pipeline-linked date columns are gated by a
`SECURITY DEFINER` RPC set (`advance_job_stage`, `move_job_back`,
`reschedule_booking`) and a guard trigger
(`private.guard_jobs_update()` in `supabase/migrations/20260718090003_functions_triggers.sql`)
that rejects direct writes to `stage`, `step`, and the pipeline-gate dates
outside those functions. This is the fix for a documented bug in the old app
(editing a date field directly bypassed stock consumption) — any change
touching these columns must preserve that guarantee.

## 1. Split `customers` out of `jobs` — AGREED

**Problem.** `jobs` embeds customer contact fields (`name`, `email`, `phone`,
`contact_method`) directly on every job row, with no relationship linking two
jobs to the same customer — not even by convention, even though `job_type =
'service'` already represents a returning customer's second job.

**Proposed schema.**

```
customers
  id            bigint generated by default as identity primary key
  name          text
  address       text        -- customer's own contact/mailing address
  phone         text
  email         text
  contact_method text
  version       integer not null default 1
  created_at    timestamptz not null default now()
  updated_at    timestamptz not null default now()
```

`jobs` changes:
- **Remove:** `name`, `email`, `phone`, `contact_method` (move to `customers`)
- **Add:** `customer_id bigint references customers(id)`
- **Keep unchanged:** `location` — this is the install site address, and is
  deliberately distinct from `customers.address` (a customer's billing/contact
  address can differ from where a system is actually installed, e.g. rental
  property, business address). Do not collapse these into one field.

**Migration consideration.** Existing trial job rows have customer data
embedded per-row with no identity linking. Splitting requires a matching pass
("which job rows belong to the same customer") by name+phone or name+email —
the same category of problem as the supplier-name matching already solved in
`scripts/import_from_export.py` (the "EnergySpurt" / "Energy Spurt" case).
Produce a dry-run match report for human review before merging any rows into a
shared customer; do not auto-merge silently.

**RLS.** New policies on `customers`: installers can `SELECT` a customer only
via a job assigned to them (join through `jobs.customer_id`) — they need
contact details to do the install. Admin: full access. No customer-facing
login exists in this system, so no customer-owned policy is needed.

## 2. `jobs.job_order` (jsonb) → real `installation_requests` table — AGREED

**Problem.** `job_order` is an unstructured jsonb blob carried over from the
old app's shape: `{ref, issued, installer, vehicle, siteAccess,
specialInstructions, systemDesc, extraNotes, savedAt, customItems[]}`. No
cross-job query surface (e.g. "which vehicle was used most often").

**Proposed schema.**

```
installation_requests
  id                    bigint generated by default as identity primary key
  job_id                bigint not null references jobs(id) on delete cascade
  job_order_ref         text
  issued_date           date
  vehicle               text        -- free text, see note below
  site_access_notes     text
  special_instructions  text
  additional_notes      text
  custom_items          jsonb       -- kept as jsonb; small ad hoc line-item
                                     -- list, not worth a child table at this
                                     -- scale — revisit only if it needs
                                     -- independent querying later
  version               integer not null default 1
  created_at            timestamptz not null default now()
  updated_at            timestamptz not null default now()

  unique (job_id)   -- see cardinality note below
```

Dropped from the old jsonb shape (superseded, not carried forward):
- `systemDesc` — duplicate of `jobs.system_description`
- `installer` (free-text name) — superseded by `jobs.assigned_installer_id`,
  a real FK to `profiles`

**IMPORTANT — what does NOT move here.** `jobs.stage`, `jobs.step`,
`jobs.assigned_installer_id`, and all pipeline-gate dates (see §3) **stay on
`jobs`**, unchanged. This was explicitly decided against an earlier draft that
would have merged them into this table. Reasons:

- `assigned_installer_id` is what the installer-visibility RLS policy checks
  directly (`assigned_installer_id = auth.uid()` on `jobs`). Moving it to a
  child table turns every job-visibility check into a join, and raises a real
  question: **can a job ever have more than one `installation_requests` row?**
  This spec assumes **1:1** (enforced by the `unique(job_id)` above) —
  confirm that assumption is still true before implementing, since it's load
  bearing for RLS.
- The guard-trigger machinery (§ above) locks and updates dates on the same
  row it checks stage/step on, in one statement. Splitting dates onto a
  second table means the three transition RPCs must touch two tables per
  transaction, and the guard's protected-column list must extend to (or be
  duplicated onto) the new table. Real work, worth doing deliberately, not by
  accident.
- `ces_submitted/received`, `rebate_submitted/received` are Stage 4
  *compliance* state, not installation logistics — they don't conceptually
  belong in an "installation request" document either way.

**RLS.** Same shape as `job_stock_items` today: visible to admin or to the
installer assigned to the parent job (via `jobs.assigned_installer_id`).
Direct writes admin-only. Unlike `jobs`, nothing here is state-machine-guarded
— a plain admin-only `UPDATE` policy should be sufficient; no RPC needed
unless that changes.

## 3. Job-date renames for planned/actual clarity — AGREED

**Problem.** The three pipeline dates already form a planned → actual-start →
actual-end sequence, but aren't named that way:

| Current name    | Set at step                    | Actually means            |
|------------------|--------------------------------|----------------------------|
| `date_booked`   | (3,1) "Date booked"            | Planned install date       |
| `install_start` | (3,3) "Install in progress"    | Actual install start       |
| `install_date`  | (3,4) "Install complete"       | Actual install completion  |

A multi-day install is already representable today — the job simply stays at
"Install in progress" until truly finished, so `install_start` and
`install_date` can already differ. The only real problem is that
`install_date` reads ambiguously (could be misread as *the* planned date
rather than *the completion* date). **Confirmed: no per-visit tracking is
wanted** — a single start/end span is sufficient, no new child table.

**Proposed — pure rename, no new columns:**
- `date_booked` → `planned_install_date`
- `install_start` → `install_started_at`
- `install_date` → `install_completed_at`

**Impact — every one of these needs updating for the rename:**
- `jobs` table definition
- `private.guard_jobs_update()` — protected-column list references these by
  name (`supabase/migrations/20260718090003_functions_triggers.sql`)
- `advance_job_stage()`, `move_job_back()`, `reschedule_booking()` — all
  reference these columns directly by name in the same file
- `app/src/lib/data.tsx` (`Job` type derives from generated types),
  `app/src/features/jobs/JobDetail.tsx`, `JobsBoard.tsx` (date displays,
  `DatePick` labels), `app/src/lib/stockCalc.ts` (`bookedInstalls()` reads
  `date_booked` for same-day clash detection)
- `scripts/import_from_export.py` — writes these column names in the SQL it
  emits
- Regenerate `app/src/types/database.types.ts` after migrating

## 4. Notification contact fields on `profiles` — AGREED

```
alter table profiles add column notification_email text;
alter table profiles add column notification_phone text;
```

Purpose: gives the existing `job_events` scaffolding (a table plus a no-op
stub Edge Function `job-events-hook`, both from Phase 1) somewhere to actually
send to, once real email/SMS notifications are built. This round only adds
the data fields — not the send logic.

## 5. Vehicle — PARKED (placement decided, full design deferred)

Electricians typically bring their own kit/vehicle; full modeling (multiple
vehicles per installer, kit tracking) is deferred to a later conversation with
Fred. **For this round:** `vehicle` is a free-text field on
`installation_requests` (§2) — **not** a `profiles` field, **not** a
normalized `vehicles` table. Revisit only when there's a concrete requirement.

## 6. Company name on `profiles` — PARKED, not added

Considered whether installers might be subcontractor companies rather than
individual employees, needing a `company_name` field. **Not added this
round** — `profiles` stays the single user identity/role table. If this
becomes real, the likely future shape is a companion `installer_details`
table (1:1 with `profiles`, admin-only), not a column bolted onto `profiles`
directly. Flagged for later, not designed further.

## 7. `stock_ces_specs.manufacturer` → real `manufacturers` table — AGREED IN PRINCIPLE, not fully speced

**Problem.** `stock_ces_specs.manufacturer` is free text, duplicated across
every stock item from the same maker. The one fact that actually matters here
— *"Deye and Jinko legal manufacturer names are verified; Sigenergy and Trina
are not"* (per `CLAUDE.md`) — is a fact about the **manufacturer's legal
name**, not about any individual stock item's spec row. Today it must be (and
was, in the Phase 2 import) set independently on every row for the same
manufacturer.

**Proposed direction (not yet fully speced — settle before implementing):**

```
manufacturers
  id            bigint generated by default as identity primary key
  name          text
  cec_verified  boolean not null default false   -- moves off stock_ces_specs
  notes         text
```

Then `stock_ces_specs.manufacturer_id bigint references manufacturers(id)`
replaces `stock_ces_specs.manufacturer text`.

**Open question, not yet answered:** does anything else need to be stored per
manufacturer (ABN, CEC listing URL, standards version)? Default to minimal
(`name`, `cec_verified`, `notes`) unless there's a concrete need.

## 8. Why two tables — `stocks` vs `stock_ces_specs` — explained, no change proposed

Recorded here because it was asked about, not because a change is proposed.
`stocks` is every inventory item, including things with no CES relevance at
all (mounting brackets, cable ties). `stock_ces_specs` is an optional 1:1
extension that only has a row for items that are actually CES-relevant
(inverters, batteries, panels) — the **absence** of a row is meaningful (it
drives the old app's "not in the CES catalogue" warning, preserved in
`app/src/features/jobs/actions.ts`'s `buildCes()`). Classic core-entity +
optional-specialized-extension pattern. Flagged by the team to reconsider
later if it stops feeling justified — no action here yet.

## Execution plan, once everything above is settled

This is a restructuring on the same scale as the original schema/RLS build
(see `/root/.claude/plans/superbase-is-going-to-woolly-book.md` for that
precedent's checkpoint structure — same discipline applies here).

1. One migration (or a short numbered series) covering: `customers` table +
   `jobs` column removal/`customer_id` addition; `installation_requests`
   table + `jobs.job_order` removal; the three date renames; `profiles`
   notification columns; `manufacturers` table +
   `stock_ces_specs.manufacturer_id`.
2. Update `private.guard_jobs_update()`'s protected-column list for the
   renamed date columns.
3. Update `advance_job_stage`, `move_job_back`, `reschedule_booking` for the
   renamed columns.
4. New RLS policies for `customers` and `installation_requests` (see notes
   above under each section).
5. Data migration for existing trial rows: customer dedup/match report
   (human-reviewed before merge), `job_order` jsonb → `installation_requests`
   row-by-row, manufacturer string → `manufacturers` row dedup.
6. Regenerate `app/src/types/database.types.ts`.
7. Frontend updates: everywhere `app/src/features/jobs/*` and
   `app/src/lib/*` currently reads `job.name/email/phone/location` directly
   now needs a join to `customers`; `job.job_order` reads move to
   `installation_requests`; date field names change throughout.
8. Re-run the same verification pattern as Phase 1: Supabase advisors, RLS
   simulation for both roles (installer sees only their assigned jobs and the
   right customer/installation-request rows via the join; guarded columns
   still rejected), full pipeline walkthrough (advance → consume → move back
   → restore) to confirm nothing regressed.
9. Update `CLAUDE.md` and the original plan doc — both currently describe the
   pre-restructure schema and will be stale in places (e.g. `CLAUDE.md`'s job
   field list, the plan doc's "Schema design" section) once this lands.
